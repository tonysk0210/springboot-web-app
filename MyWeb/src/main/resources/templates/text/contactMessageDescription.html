聯絡訊息分頁與排序管理 — Spring Data JPA 與 Thymeleaf 整合應用

View Contact Page 頁面僅顯示 status 為 OPEN 的 Contact 物件，並由 AdminController 負責處理對 /viewContactMessage/page/{currentPageNum} 的請求。

系統利用 Spring Data JPA 的內建分頁 Pagination 和 Sorting 機制，從 contact_msg 資料表建立 Page<Contact> 物件。 接著，從中擷取當前頁面的 List<Contact>　資料，並透過 Thymeleaf 將結果渲染至前端。

當使用者透過分頁或欄位排序連結進行操作時，系統會根據新的 URL 參數更新查詢條件，重新建立分頁結果並渲染畫面，提供完整且互動性良好的分頁與排序功能。

🔧 建立 Pageable 與 Page<Contact>

為建立 Page<Contact> 物件，必須先建立 Pageable 物件，用來定義欲查詢的分頁條件，當中包含了以下屬性：

  • pageNumber：目前頁碼
  • sizePerPage：每頁顯示的筆數
  • sortField：排序欄位
  • sortDir：排序方向（asc / desc）
Pageable pageable = PageRequest.of(
    currentPageNum - 1,   // Spring Boot 的分頁機制為 zero-based（從 0 開始），因此若前端傳入的頁碼從 1 起算，需將其減 1 以符合系統預期。
    sizePerPage,
    sortDir.equals("asc") ? Sort.Direction.ASC : Sort.Direction.DESC,
    sortField
);

這些參數可透過 URL 指定，例如：

/admin/viewContactMessage/page/1?sortField=name&sortDir=asc

系統會根據這些參數建立對應的 Pageable 物件，並傳入contactRepository查詢方法中，以取得特定狀態（status = OPEN）的聯絡訊息。 

Page<Contact> pageOfContacts =
    contactRepository.findByStatusWithPageableAtQuery(
        ProjectConstant.STATUS_OPEN,  // 查詢 status 為 OPEN 的聯絡訊息
        pageable);  // 套用分頁與排序條件

// 查詢所有 status 為 OPEN 的 Contact 物件並封裝成一個 Page<Contact>

接著，透過 .getContent() 方法從 pageOfContacts 中擷取當前頁面的資料內容（型別為 List<Contact>），並傳遞至前端進行渲染。 

List<Contact> listOfContactsPerPage = pageOfContacts.getContent(); // 從 Page 物件中取得當前分頁的資料列表 (List<Contact>)，並傳送至前端

使用者亦可透過分頁與排序連結（例如：表格標題欄中帶有參數的超連結），動態調整 Pageable 條件，實現彈性且友善的聯絡訊息管理與檢視體驗。

聯絡訊息關閉流程與頁面重導機制

前端的 「Close」 按鈕會附帶 contactId 及當前的分頁參數（如 pageNum、sortField、sortDir），確保完成關閉操作後能正確導回原頁面，維持良好的使用者瀏覽體驗。

當管理者點選 「Close」 後，系統會依據 contactId 取得該筆 Contact，並將其 status 設為 CLOSED，再執行儲存。

最後進行 redirect 返回聯絡訊息頁面，並保留分頁與排序參數，重新查詢並渲染更新後結果。
