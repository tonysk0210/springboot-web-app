個人資料更新流程與雙重驗證機制說明（格式驗證 + 帳號唯一性）

使用者進入 /profilePage 時，系統會透過 ProfilePageController 從 HttpSession 中取得目前登入的 Person ("loggedInPerson") 物件， 並將其屬性複製至 Profile 物件。再透過 Model 將 Profile 傳遞至前端表單，用於資料顯示與修改。待表單提交後，系統將執行兩層驗證流程：資料格式驗證與 Email 唯一性檢查。

✅ 第一層：欄位格式驗證（由 Spring Boot 驗證框架處理）

表單提交後，資料將綁定至 Profile 物件，並由後端進行驗證與帳號唯一性檢查。

  • name：⚠️ Name must not be blank
  • mobile：
    • ⚠️ Mobile number is required
    • ⚠️ Mobile Number must be exactly 10 digits long
  • email：⚠️ Email is required
  • address1：⚠️ Address1 is required
  • city：⚠️ City is required
  • zipcode：
    • ⚠️ Zip code is required
    • ⚠️ Zip code must be exactly 3 digits

✅ 第二層：帳號唯一性驗證

系統將檢查 Email 是否與其他帳號重複，避免使用者更新為已註冊的 Email： 

// 檢查資料庫中是否已存在其他使用者使用相同的 email（但不是目前這位使用者）
if (personRepository.existsByEmailAndPersonIdNot(profile.getEmail(), person.getPersonId())) {
    result.rejectValue(
        "email",                                     // 指定要綁定錯誤的欄位為 email
        "profile.email.exists",                      // 錯誤訊息代碼，可對應 messages.properties
        "An account with email " + profile.getEmail() + " already exists." // 預設錯誤訊息內容
    );
    return "authenticated/profile"; // 回傳至個人資料頁面並顯示錯誤
}

若Email已重複存在顯示 ⚠️ An account with email " + profile.getEmail() + " already exists. 若無重複，系統則會同步更新 Spring Security 中的 Authentication 資訊，將 principal 更新為新的 Email。

更新與稽核：保存最新個人資料並同步 Session 狀態

修改完成後，系統會透過 personRepository.save(person) 將更新後的 Person 物件儲存至資料庫。此時，AuditorAware 會在執行資料更新（UPDATE）時，依據更新後的Authentication 自動填入 updatedBy 和 updatedAt 欄位，完成使用者資料的稽核更新。 此外，為確保後續操作中使用的是最新的使用者資訊，系統亦會同步更新 HttpSession 中的 Person ("loggedInPerson") 物件，以反映此次變更。 最後，系統會重新導向至 redirect:/profilePage?updated=true，由前端畫面根據 URL 參數顯示成功訊息：
Profile updated successfully. 

Person savedPerson = personRepository.save(person); // 將更新後的 Person 實體儲存至資料庫

session.setAttribute("loggedInPerson", savedPerson); // 同步更新 Session 中的 loggedInPerson 物件
