方案管理頁面：載入與新增 Plan 的資料處理流程

當管理員進入Plan Management頁面時(admin/planPage)，系統會透過下列方法，自動查詢所有已建立的所有方案資料： 

// 此方法會在每次進入 AdminController 所控制的頁面時自動執行
@ModelAttribute("planList")
public List<Plan> displayAllPlans() {
    return planRepository.findAll();
}

上述方法將所有 Plan 物件以 List<Plan> 型態傳遞至前端進行渲染。前端頁面將顯示所有方案資訊，並為每筆資料提供「VIEW」 與 「DELETE」 操作，每個按鈕皆附帶對應的 planId 作為查詢參數。

此外，系統會提供一個空的 Plan 物件，作為 「Add Plan」 表單的綁定對象。一旦表單送出後 Plan 物件將會觸發後端驗證。 name : ⚠️ Plan name is required

若驗證通過，系統將該方案儲存至資料庫中的 plan 資料表( planRepository.save(plan); )，並執行 redirect:/admin/planPage 導回當前頁面。此時，displayAllPlans() 方法會再次執行，重新載入並渲染最新的方案清單。

刪除 Plan 前的關聯解除處理

當管理員執行刪除操作時，系統會根據傳入的 planId，透過下列程式碼查詢對應的 Plan 物件：

Optional<Plan> planToBeDeleted = planRepository.findById(planId);

根據 Plan 類別中的 JPA 設定：

// mappedBy 表示此關聯的擁有端為對方實體（此例為 Person）中的 plan 屬性，外鍵實際存在於 Person 資料表中。
@OneToMany(mappedBy = "plan", fetch = FetchType.LAZY, targetEntity = Person.class)
private Set<Person> persons;

此設定表示 Person 是擁有端（owning side），Plan 為被參照端（inverse side）。 在 JPA 中，外鍵實際存在於 Person 資料表，因此必須由 Person 主動維護與 Plan 的關聯關係。

🚨 刪除 Plan 前必須先解除所有 Person 的關聯：

若直接刪除仍有關聯的 Plan 物件，將會違反資料庫的外鍵約束而產生例外。 因此，系統應先將每一位已關聯的 Person 的 plan 屬性設為 null，如下： 

for (Person person : plan.getPersons()) {
    person.setPlan(null);
    personRepository.save(person); // 儲存解除關聯後的變更
}

完成關聯解除後，即可安全地刪除該 Plan：

planRepository.deleteById(planId);

此流程可確保刪除操作符合 JPA 的關聯維護規範，避免因資料表間的外鍵依賴而導致錯誤發生。
